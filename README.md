# AI vs Employee ROI Calculator

A standalone web app for business decision makers. Plug in your numbers; see day-by-day cumulative cost of building AI vs keeping (or hiring) an employee, plus a phased rollout playbook generated by Gemini.

## Models four scenarios simultaneously

1. **Replace 1 employee** — direct salary saved by AI taking over a role
2. **Augment a team** — % of manual work removed across N people
3. **Avoid future hire** — headcount cost not added because AI handles the load
4. **Reduce errors** — operational rework cost prevented

The chart shows cumulative AI cost vs. employee baseline cost, with a break-even reference line. The playbook is an LLM-generated 6–8 phase rollout (Week 1 → Month 6+) with cost, owner, and risks per phase.

## Stack

- Next.js 14 (App Router) + TypeScript
- Tailwind CSS (with dark mode)
- Recharts for the burndown chart
- `@google/generative-ai` (Gemini 2.5 Flash) for the playbook
- `zod` for LLM JSON validation
- Vitest for math unit tests

## Setup

```bash
npm install
cp .env.example .env.local       # then add your GEMINI_API_KEY
npm run test                      # math unit tests
npm run dev                       # http://localhost:3000
```

If you don't set `GEMINI_API_KEY`, the `/api/playbook` route will fall back to a built-in sample playbook. You can also click **Use sample** in the UI.

## Notes on the math

- Build cost is paid up front on day 1. Run cost = `monthlyOpex × 12 / 365` plus a maintenance FTE allocation.
- Replace, augment, and error-reduction savings include a linear ramp from 0 to full value over `rampUpDays` (default 60).
- Avoid-hire is a step function turning on at `avoidedHireMonth × 30` days.
- Break-even is the first day where cumulative scenario savings ≥ cumulative AI cost.
- NPV discounts each day's net cashflow at `(1 + discountRate)^(day/365)`.

The LinkedIn-post anchor numbers ($72K productivity gain on $180K team, $60K avoided hire, $80K AI cost) reproduce in the calculator at `productivityGainPct ≈ 0.286` with `benefitsMultiplier = 1.4` — the post implied a lower benefits multiplier than the default.
